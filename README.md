# skill-evaluator

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE) | [English](README_EN.md)

一个用于评估 AI Agent Skill 质量的 Skill。它会从 8 个维度给 Skill 加权打分，找出不足，给出改进建议，并支持多个 Skill 横向对比。

版本 3.0.0 移除了外部模型 API 路线。多模型评估只使用当前 Agent 平台原生支持的模型；如果环境不支持原生多模型，则自动使用严格派、务实派、专家派三视角评估。

## 特性

- **八维评分**：每个维度 0-10 分，并按权重计算总分
- **等级映射**：自动映射为 S/A/B/C/D 等级
- **改进建议**：对薄弱维度给出问题、影响和具体建议
- **多 Skill 对比**：并列评分、优劣势分析、经验总结和排名
- **原生多模型交叉验证**：多个平台原生模型独立评分、互审，再由仲裁者综合
- **多视角回退**：原生多模型不可用时，使用严格派、务实派、专家派三视角评估
- **零外部凭据**：不需要配置或保存任何凭据，不调用外部模型 API

## 评估维度

| 维度 | 权重 | 说明 |
|------|------|------|
| D1. 元数据质量 | 15% | `name` 和 `description` 是否清楚、准确、可触发 |
| D2. 执行引导清晰度 | 15% | Agent 加载后是否知道该怎么做 |
| D3. 领域知识密度 | 15% | 是否提供通用 Agent 不具备的专业知识 |
| D4. 工作流完整性 | 15% | 是否有清晰的端到端流程、分支和异常处理 |
| D5. 输入输出清晰度 | 10% | 用户是否知道该提供什么、能得到什么 |
| D6. 资源利用 | 10% | 是否合理使用 `references/`、`scripts/`、`assets/` |
| D7. 写作质量 | 10% | 结构、表达和可读性是否适合 Agent 执行 |
| D8. 范围与聚焦 | 10% | 是否专注解决一类清楚的问题 |

## 等级标准

| 分数 | 等级 | 含义 |
|------|------|------|
| 9.0-10 | S | 标杆级 |
| 7.5-8.9 | A | 优秀，可直接使用 |
| 6.0-7.4 | B | 可用，但有明显改进空间 |
| 4.0-5.9 | C | 较弱，需要大幅修改 |
| 0-3.9 | D | 存在根本问题，建议重做 |

## 安装

```bash
git clone https://github.com/sunxingboo/skill-evaluator.git ~/.claude/skills/skill-evaluator
```

也可以将 `SKILL.md` 和 `references/` 目录复制到你的 Agent Skill 目录。

## 使用示例

评估单个 Skill：

```text
帮我评估一下 my-skill 这个 Skill
```

对比多个 Skill：

```text
对比 skill-a 和 skill-b，哪个设计更好
```

按路径评估：

```text
评估 ~/projects/my-skill/SKILL.md
```

使用交叉验证：

```text
用多模型交叉验证评估 my-skill
```

使用多视角模式：

```text
用多视角模式评估 my-skill
```

## 执行策略

### 策略 A：原生多模型

当当前 Agent 平台支持多个原生模型时：

1. 每个模型独立评估目标 Skill。
2. 每个模型审查其他模型的评分。
3. 主 Agent 汇总分歧并仲裁最终分数。

### 策略 B：串行多视角

当原生多模型不可用、原生模型不足，或用户明确要求多视角时：

1. 严格派评审关注规范、边界和遗漏。
2. 务实派评审关注可用性、清晰度和实际执行体验。
3. 专家派评审关注专业深度、资源设计和范围控制。
4. 最后统一比较分歧并仲裁。

## 移除外部 API 后的影响

正面影响：

- 不需要配置凭据。
- 不会保存凭据。
- Skill 更轻、更容易审计。
- 后续使用更稳定，不依赖单独的外部服务。

取舍：

- 不再通过外部 API 调用额外模型。
- 如果当前环境不支持原生多模型，交叉验证会退化为单模型多视角评估。
- 结果的来源多样性会降低，但可维护性和安全性更好。
