# skill-evaluator

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE) | [English](README_EN.md)

项目地址：`git@github.com:Mysqi/skill-evaluator.git`

一个用于评估 AI Agent Skill 质量的 Skill。它会从 8 个维度给 Skill 加权打分，找出不足，给出改进建议，并支持多个 Skill 横向对比。

版本 3.0.0 移除了外部模型 API 路线。多模型评估只使用当前 Agent 平台原生支持的模型；如果环境不支持原生多模型，则自动使用严格派、务实派、专家派三视角评估。

## 特性

- **八维评分**：每个维度 0-10 分，并按权重计算总分
- **等级映射**：自动映射为 S/A/B/C/D 等级
- **改进建议**：对薄弱维度给出问题、影响和具体建议
- **多 Skill 对比**：并列评分、优劣势分析、经验总结和排名
- **原生多模型交叉验证**：多个平台原生模型独立评分、互审，再由仲裁者综合
- **多视角回退**：原生多模型不可用时，使用严格派、务实派、专家派三视角评估
- **零外部凭据**：不需要配置或保存任何凭据，不调用外部模型 API

## 评估维度

| 维度 | 权重 | 说明 |
|------|------|------|
| D1. 元数据质量 | 15% | `name` 和 `description` 是否清楚、准确、可触发 |
| D2. 执行引导清晰度 | 15% | Agent 加载后是否知道该怎么做 |
| D3. 领域知识密度 | 15% | 是否提供通用 Agent 不具备的专业知识 |
| D4. 工作流完整性 | 15% | 是否有清晰的端到端流程、分支和异常处理 |
| D5. 输入输出清晰度 | 10% | 用户是否知道该提供什么、能得到什么 |
| D6. 资源利用 | 10% | 是否合理使用 `references/`、`scripts/`、`assets/` |
| D7. 写作质量 | 10% | 结构、表达和可读性是否适合 Agent 执行 |
| D8. 范围与聚焦 | 10% | 是否专注解决一类清楚的问题 |

## 等级标准

| 分数 | 等级 | 含义 |
|------|------|------|
| 9.0-10 | S | 标杆级 |
| 7.5-8.9 | A | 优秀，可直接使用 |
| 6.0-7.4 | B | 可用，但有明显改进空间 |
| 4.0-5.9 | C | 较弱，需要大幅修改 |
| 0-3.9 | D | 存在根本问题，建议重做 |

## 安装

```bash
git clone git@github.com:Mysqi/skill-evaluator.git ~/.claude/skills/skill-evaluator
```

也可以将 `SKILL.md` 和 `references/` 目录复制到你的 Agent Skill 目录。

## 使用示例

评估单个 Skill：

```text
帮我评估一下 my-skill 这个 Skill
```

对比多个 Skill：

```text
对比 skill-a 和 skill-b，哪个设计更好
```

按路径评估：

```text
评估 ~/projects/my-skill/SKILL.md
```

使用交叉验证：

```text
用多模型交叉验证评估 my-skill
```

使用多视角模式：

```text
用多视角模式评估 my-skill
```

## 执行策略

### 策略 A：原生多模型

适用于当前 Agent 平台可以同时调用两个或更多原生模型的情况。它的目标是让多个模型先独立判断，再互相指出分歧，最后由主 Agent 给出统一结论。

```text
用户请求多模型评估
        ↓
确认当前平台可用的原生模型
        ↓
多个模型并行独立评分
        ↓
模型之间互相审查明显分歧
        ↓
主 Agent 汇总证据并仲裁
        ↓
输出最终分数、共识度和改进建议
```

执行时分三步：

1. **并行独立评分**：每个可用模型读取同一份 Skill 内容，按 8 个维度分别打分，并给出证据。
2. **交叉审查分歧**：重点检查相差 2 分或以上的维度，要求模型说明为什么同意或不同意其他评分。
3. **统一仲裁结论**：主 Agent 根据证据决定最终分数，并标记每个维度是“一致”“多数”还是“仲裁”。

### 策略 B：串行多视角

适用于原生多模型不可用、可用模型不足，或用户明确要求多视角评估的情况。它不依赖额外模型，而是让同一个 Agent 从三个不同角度连续评估。

```text
无法使用原生多模型
        ↓
严格派评审：看规范、边界和遗漏
        ↓
务实派评审：看清晰度、上手难度和实际可用性
        ↓
专家派评审：看专业深度、资源设计和范围控制
        ↓
比较三个视角的评分差异
        ↓
输出最终分数、分歧说明和改进建议
```

三个视角的关注点不同：

| 视角 | 重点问题 |
|------|----------|
| 严格派 | 触发条件是否准确，边界是否清楚，异常情况是否覆盖 |
| 务实派 | Agent 是否容易照着执行，用户是否容易理解输入和输出 |
| 专家派 | 是否真的提供了专业知识，资源组织是否合理，范围是否克制 |

最后会把三个视角的评分放在一起比较。若某个维度差异较大，就说明分歧原因，并给出最终仲裁分数。

## 项目来源与取舍

本项目基于原始项目二次修改：<https://github.com/sunxingboo/skill-evaluator>

当前版本保留 Skill 评估、多 Skill 对比、原生多模型评估和串行多视角评估；同时移除了外部模型 API 路线。

这样做的影响：

- 不需要配置或保存任何凭据。
- 不依赖单独的外部服务。
- 如果当前环境不支持原生多模型，交叉验证会自动退回串行多视角。
- 结果的模型来源多样性会降低，但安装、使用和维护更简单。
